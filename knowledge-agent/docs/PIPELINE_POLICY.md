# 파이프라인 설계 정책 (world-model 구축)

> 이 문서는 **"왜 이렇게 정했나"**를 기록한다. "무엇을 했나"는 코드를 보면 되지만,
> "왜"는 코드에 안 남으므로 여기 박아둔다. 다음 세션 복귀 + 면접 서사용.
>
> 기준 시점: ReAct 논문(2210.03629) 한 편을 IA → align까지 통과시킨 직후.
> (참고: `docs/KARMA_AGENTS.md`는 구버전 설계 기준이라 현재와 불일치 — 갱신/폐기 대상)

---

## 0. 프로젝트 전체 그림

개인 지식 시스템, 3단계. 지금은 ①을 구축 중.

```
① world-model 구축   논문들을 읽어 "분야 지식 지도"를 자동 생성   ← 현재
② self-model         그 지도에서 내가 뭘 아는지(mastery) 측정
③ 추천               커버 안 된 곳 기반 학습 추천
```

①은 KARMA(논문→지식그래프 자동구축)의 파이프라인을 참고하되, 우리 목적
("정확한 분야 그래프 완성"이 아니라 **"내 커버리지 지형도"**)에 맞게 가지치기·변형한다.

핵심 철학: **맵은 지형도지 디테일 저장소가 아니다.** 내가 아는 영역/모르는 영역을
표시하는 게 목적이고, 디테일하게 파고드는 건 그다음(self-model 카드 → 프론티어 LLM) 몫.
이 한 가지가 아래 거의 모든 결정을 좌우한다.

---

## 1. 파이프라인 흐름

```
PDF
 │
 ▼ 01 IA   (Ingestion)         PDF 파싱(pymupdf4llm) + 청소(references·표·그림 제거)
 ▼ 02 RA   (Reader)            ## 헤더로 섹션 분할 (관련성 컷은 보류)
 ▼ 03 SA   (Summarizer)        섹션별 "개념 위주 산문" 요약 (수치·세팅 버림)
 ▼ 04 EEA  (Entity Extraction) 개념 + 정의(다중) 추출
 ▼ 05 align (정규화)             개념이 기존 맵에 있나/새것인가 판정 (merge/new)
 ▼ 06 REA  (Relation Extraction) 노드 쌍 관계 판정 (is_a/part_of) — 판정만
 ▼ 07 CRA  (Conflict Detection)  새 엣지 vs 맵 엣지 모순 감지 — 감지만
 ▼ 08 통합 (Integration)          05·06·07 받아 맵에 반영 (노드·엣지 **누적**)
```

> **번호 재정비(2026-06):** 구 07 SAA(schema alignment) 폐기, CRA를 07로, 통합을 08로.
> SAA 폐기 이유 = 관계 어휘 정렬은 06이 이미 하고, 노드 `type` 배정은 소비처가 없으며
> (mastery·경로추천 어느 것도 type 미사용), 맞출 고정 스키마 자체가 없음(라벨 없는 분야를
> 맨바닥부터 구축). `type` 필드는 남기되(시드엔 있고 신규는 null) 전담 에이전트는 없앰.
> 구 `depends_on` 관계도 완전 폐기 — 선수지식 경로는 is_a 계층으로 유도(텍스트에 명시 안 됨).

- 파일: `agents/agent_01_ingestion.py` ~ `agent_08_integration.py`
- 프롬프트: `agents/prompts/` (영어 본문 + 한글 1:1 번역 주석. import 시 영어만 따라옴)
- 산출물: `data/parsed/{arxiv_id}/{arxiv_id}_0N.*` (논문별 폴더 + 단계 번호)

## 2. 단계별 정책

### 01 IA — 파싱·청소

- **역할:** PDF → 깨끗한 본문 마크다운 + 메타데이터(arxiv_id).
- **결정:**
  - 파싱은 `pymupdf4llm.to_markdown(use_ocr=False)` (검증된 라이브러리 원칙, 직접 안 짬).
  - `## REFERENCES` 헤더 이후 통째 삭제 → references + appendix 동시 제거.
  - 표(`|`로 시작하는 행)·그림 잔해(`picture text` 블록, `intentionally omitted`)·페이지 가구(4회+ 반복되는 짧은 줄 = 러닝 헤더, 페이지번호만 있는 줄) 규칙 제거.
- **근거:**
  - 표·그림이 깨지는 건 버그가 아니라 **무료 필터** — 거기 담긴 건 수치·시각자료라 개념맵에 불필요. 비전 모델로 표 살리는 건 인프라 과잉투자.
  - references 컷은 의미 판단이 필요 없는 규칙 작업 → IA에서 싸게. (KARMA는 RA 관련성 컷에 맡기지만, 우린 그 컷을 보류했으므로 IA로 당김.)
  - **함정:** "appendix" 단어가 본문에 흩어져 있어 단어 매칭 금지 → 헤더 줄(`## REFERENCES`)만 경계로.
- **KARMA 대비:** KARMA IA는 LLM으로 OCR 오류 정규화(생의학 PDF가 지저분해서). 우리 arxiv PDF는 깨끗해서 규칙 변환으로 충분 → 사실상 "전처리". 입력이 거칠어지면(코드베이스 등) LLM 정규화로 진화 가능.
- **주의:** 청소 규칙은 pymupdf4llm 출력 모양에 **종속**. 파서 바꾸면 재작성. 새 논문은 `inspect_pdf.py`로 헤더 모양 먼저 점검.

### 02 RA — 섹션 분할

- **역할:** 본문을 `## 헤더`마다 세그먼트로 분할 (ReAct는 14개).
- **결정:** **분할만 한다. 관련성 컷(δ)은 보류.**
- **근거:**
  - KARMA RA는 분할 + 관련성 점수(세그먼트가 그래프에 유의미한가) 컷. 우리는 컷을 **Collector(자동 수집) 단계로 미룸** — 컷 설계는 "입력이 얼마나 잡음 많나"에 달렸는데, 그건 수집기가 정해져야 알 수 있음. 지금 정하면 인프라 과잉투자.
  - 분할은 LLM 안 씀 — pymupdf4llm이 `## 헤더`를 잘 보존해서 규칙으로 충분. (KARMA는 LLM 분할이지만 그건 PubMed PDF가 헤더 구조를 안 남겨 의미로 복원해야 했기 때문.)
- **함의:** RA가 지금 "JSON 변환"처럼 가벼운 건 ① 관련성 컷을 뺐고 ② 헤더가 깨끗해 LLM 불필요하기 때문. 누락이 아니라 두 결정의 결과.

### 03 SA — 요약

- **역할:** 각 섹션을 "개념과 그 연결이 드러나는 짧은 산문"으로 압축. (파이프라인에서 처음 LLM 등장.)
- **결정:**
  - **수치·실험세팅·하이퍼파라미터 버림** (EM 35.1, PaLM-540B, 32-shot 등). 개념의 정체성에 필요한 수치만 남김.
  - **외부지식 절대 금지** — 입력에 있는 것만. LLM 사전지식으로 채우면 할루.
  - **산문으로** (불릿·트리·계층 목록 금지). 개념을 문장으로 *연결*해 서술 (관계 판정은 안 함, 그건 다음 단계).
  - 길이 고정 안 함 — 개념 밀도가 정함. 분량 맞추려 개념 빼기 금지.
- **근거:**
  - 맵이 "지형도"라 수치는 노이즈. (KARMA SA는 정반대로 수치 verbatim 보존 — KARMA 맵은 수치가 곧 지식이라서. **목적이 프롬프트를 정반대로 만든 대표 사례.**)
  - KARMA "Summarizer 빼면 정확도 22.9%↓"는 요약 단계의 가치지, Reader/Summarizer 분리의 근거가 아님. 우리는 한 콜로 통합(Reader 역할이 이미 RA·evaluator에서 소진됨).
- **삽질 기록:** 처음 "concept map"이라 했더니 LLM이 시각적 트리를 그림. "concise prose" + 좋은 예시 하나로 교정. **프롬프트 단어 하나가 출력 형식을 좌우** — 예시가 금지 나열보다 강력.

### 04 EEA — 개념 추출

- **역할:** 요약문에서 개념 + 정의 추출. 출력: `{name, definitions: [...]}` 리스트.
- **결정:**
  - **개념 = 고유 이름 가진 것** (방법·기법·메커니즘·벤치마크). 속성구·막연한 지칭·일반 명사구·모델 인스턴스(PaLM, GPT-3)는 제외. 판별 테스트: "서베이 색인에 이름 항목으로 실릴 만한가?"
  - 정의는 **개념 자체**가 뭔지 (이 논문에서의 적용·대조 말고). 외부지식 금지.
  - **정의를 여러 개 보관** (다중 정의). 같은 이름은 묶되 정의는 안 버리고 리스트로.
  - **기존 맵은 안 봄** (그건 align). EEA는 "이 논문에 이런 개념들이 있다"만.
- **근거 — 폐기한 것들 (중요):**
  - **status 플래그 폐기.** "이 개념을 논문이 정의했나(defined)/사용만 했나(used)"를 LLM이 판정하게 했으나 **불안정 + 과부하.** (LLM에 추출+정의+메타판정 3중.) 분류는 align의 측정(임베딩 거리)에 맡기는 게 정직. → "이름값이 아닌 적합성으로" 원칙.
  - **빈 문자열/필드 유무로 상태 표현 금지** — 부재로 의미 싣기는 버그와 정상을 구분 못 함.
  - **aliases 폐기** — 약어 병합 보조용이었으나, ① 이름 비교를 안 하기로 했고 ② 소비처가 없어 안 씀. (필요해지면 그때.)
  - **"긴 정의 택하기" 폐기** — 같은 개념도 섹션마다 정의 품질이 다름(ReAct 12개 출현: 깨끗한 것~GPT-3 코드 오염까지). 길이 ≠ 품질, 오히려 맥락 붙으면 길어짐(=오염). 좋은 정의가 삭제될 위험 → **안 버리고 다 모아 align이 거리로 best 선택.**
- **dedup(group) 현재 역할:** 같은 이름(대소문자) 묶어 정의 리스트로 모으기만. 판단(같은 개념인가, 어느 정의가 best인가)은 align으로.

### 05 align — 정규화 (핵심 단계)

- **역할:** 각 개념이 기존 맵에 있나(merge)/새것인가(new) 판정. **판정만, 맵엔 안 씀** (실제 반영은 08 통합).
- **결정 — 2단계 (임베딩 후보 + LLM 확정):**
  1. **임베딩으로 후보 추림** (싼 거친 필터): 개념의 모든 정의 × 시드맵 노드 정의문 코사인 유사도. 상위 K(=2) 중 floor(=0.40) 넘는 것만 후보. 무관한 건(FEVER 0.22) 즉시 new.
  2. **LLM으로 확정** (정밀): 후보만 "이 둘 같은 개념?" 판정 (SAME/DIFFERENT).
- **LLM 판정 기준:**
  - SAME = "한 가지를 가리키는 두 이름" (약어 CoT=chain-of-thought, 동일 의미).
  - DIFFERENT = **변형·확장·발전형** (zero-shot CoT vs CoT), 특정 방법 vs 일반 개념 (STaR vs reasoning), 단순 관련/결합.
- **근거:**
  - **정의문으로 비교 (이름 X).** KARMA는 이름 임베딩(생의학은 도메인 임베딩 모델 SapBERT 등으로 이름만으로 식별 가능). AI 개념은 그런 모델이 없고 이름만으론 의미가 안 잡혀 → 정의문 임베딩. **이 선택이 "정의 어디서 오나/품질" 문제를 새로 만든, 우리만의 영역.**
  - **임베딩만으론 부족** — 같은 개념도 정의 표현 차이로 0.6대, 무관한 것도 0.58 → 점수가 안 갈려 단일 임계값 불가. 그래서 임베딩=거친 필터, LLM=의미 판정으로 역할 분담. (KARMA의 임베딩 후보 + LLM verification 다단계 정신과 일치.)
  - 이게 **핵심 기술 부채 ①("이름만 겹치면 억지연결")의 해소.** LLM이 점수가 아니라 의미로 판단 → 점수 낮은 진짜(CoT 0.417)는 살리고, 점수 높은 가짜(STaR 0.583→reasoning)는 거름.
- **검증:** 시드맵 9노드에 merge/new/boundary 의도를 심어 채점. CoT 계열만 merge, 변형·일반어 끌림·무관은 전부 new로 정확히 갈림.

### 06 REA — 관계 추출

- **역할:** 05에서 판정한 노드들 사이에 관계(엣지)를 긋는다. 점 다 찍었으니 선.
  **판정만 한다 — 실제 맵엔 08 통합에서 쓴다.**
- **결정:**
  - **2단계 구조.** 1단계(자유 서술: A를 문법상 주어로 한 문장 → 방향을 문장구조에
    인코딩) → 2단계(`is_a`/`part_of`/`none` 매핑). 3종에 안 맞는 건 버리지 않고
    **예외 로그**로 남긴다.
  - **REA 스코프 = `is_a` + `part_of`만.** 화이트리스트도 이 2종(`depends_on` 폐기).
    선수지식/uses/확장류는 none으로 떨어져 예외 로그에 쌓인다.
  - **양방향.** 쌍마다 A→B, B→A 둘 다 1단계에 던진다(역전 저주 대비). 양쪽이
    동시에 라벨을 받는 모순은 지금 안 거르고 CRA(07)로 미룸 — REA는 긋기만.
  - **self-pair 차단.** merge로 끝점(`matched_node`)이 같아진 쌍은 자기루프라 스킵.
  - **근거 저장(빚 ④).** 채택 엣지·예외 둘 다에 heading + 1단계 문장 + rationale.
  - **페어링 = 이름 substring 매칭** (같은 섹션 요약문에 함께 등장한 노드끼리).
    05에 섹션별 provenance가 없어 이름 매칭으로 근사. 과생성돼도 REA가 판정하니 OK.
  - **프롬프트 분리.** `prompts/p06_relations_stage1.py`, `p06_relations_stage2.py`
    (영어 본문 + 한글 1:1 주석. 파일명 `p` 접두 = 숫자 시작 모듈 import 불가 회피).
- **근거:**
  - **`depends_on` 폐기.** 선수지식은 한 논문 텍스트가 주는 게 아니라 **맵 위상**에서
    떠오른다(정의 의존 / is_a 상향 / 공출현 비대칭). REA가 한 편에서 짜내면 방향오류·
    노이즈만 생김. → 관계를 별도 엣지로 두지 않고 **is_a 계층으로 유도**(빚 ⑤와 합류).
    화이트리스트에서도 제거(2종 확정).
  - **2단계인 이유 = 예외 로그.** "3종 중 택1"만 시키면 LLM이 억지로 끼워맞춰 갭이
    안 보인다. 자유 서술 후 매핑 실패를 로그로 남겨야 "3종 충분성"을 데이터로 본다.
    (리서치: 화이트리스트만으론 스키마 밖 관계가 새어 어차피 하드 필터 필요 — 그
    필터를 *조용히 버리지 말고 로그로* 남기는 게 이 설계.)
- **실측 (ReAct 1편, 70쌍, 양방향 → self-pair 제외 134 방향):**
  - **채택 엣지 = 1** (`CoT-SC is_a chain-of-thought`). is_a/part_of는 한 논문 안에
    거의 안 나옴 = **정상**. 논문은 자기 방법을 *분류*하지 않고 *비교·개선*하기 때문.
    is_a 골격은 여러 논문이 공통 상위 개념을 공유하며 **enrichment로** 채워진다.
  - **예외 132개.** rationale 빈도: enhance 38 / utilize 16 / contrast 11 /
    build 10 / apply 10 / integrate 9 / use·uses 10 ... → **none의 ~70%가
    uses/enhance 계열로 수렴.**
  - **판정: 3종 충분.** uses/enhance는 재발(동결 원칙의 "재발" 조건 충족)하지만
    **목적상 제외가 맞다** — 경로(depends_on)에 노이즈, 엣지 폭발, 의미 불안정.
    그 정보는 노드 정의(`definitions[]`) + 예외 로그에 이미 보존돼 *안 버려진다*.
    a priori 논쟁이 아니라 70쌍 실측으로 닫힘.
- **함정/삽질:**
  - **self-pair 버그.** 첫 실행 edges 6 중 5개가 CoT 변종 자기루프(`CoT`,
    `Chain-of-Thought`, `... (CoT) prompting`이 05에서 전부 같은 시드 `chain-of-thought`
    로 merge됨 → 끝점이 같아짐). → endpoint 같으면 스킵으로 수정. (edges 6→1)
  - **"advancement" 누수.** "advancement of"가 is_a로 샘. 단어 매칭으론 못 막음 →
    stage2 규칙에 명시(extends/based on/advancement/enhancement = none). 경계 케이스는
    가끔 새는 게 정상; 누수율은 한 바퀴 더 보고 판단.
- **산출물:** `{id}_06.stage1.json`(1단계 날것) / `_06.relations.json`(채택 엣지) /
  `_06.exceptions.json`(none + rationale).
- **KARMA 대비:** KARMA REA = 정규화 쌍 + 확률 `p(r|·)` + 임계값 θ_r, multi-label,
  관계 열림. 우리 = 2종 닫음 + 2단계 + 예외 로그. **구 SAA가 할 "관계 정규화"를 REA
  2단계가 이미 한다** → SAA를 폐기할 수 있던 이유 중 하나.

### 07 CRA — 모순 감지 (Conflict Detection)

- **역할:** 새 논문 엣지(06)가 기존 맵 엣지와 **모순**되는지 감지. **감지만 한다 —
  엣지를 빼거나 바꾸지 않음.** 반영/보류는 08 통합 몫. LLM 안 씀(끝점 비교 = 순수 코드).
- **모순 규칙 (감지 키 = `(from, to)` 쌍):**
  - `rel_mismatch` : 같은 `(from, to)`인데 rel이 다름 (is_a vs part_of).
  - `cycle` : 방향 뒤집힌 같은 rel 동시 존재 (A is_a B & B is_a A).
  - **모순 아님:** `(from, to)`가 다르면 정상. 예) "ReAct is_a 기법" + "ReAct part_of
    시스템" → 끝점이 달라 둘 다 참 가능. **절대 충돌로 잡지 않음.**
- **두 방향을 본다:** 새 vs 기존 맵(보류는 새 엣지만) + 새 vs 새(동일 논문 내, 둘 다 보류).
- **`resolve()`는 스텁(미구현).** 충돌난 둘 중 뭐가 맞는지 판정하는 건데, 실제 충돌
  사례가 아직 0건(엣지 1개)이라 방식을 정할 데이터가 없다. 충돌이 쌓이면 결정
  (사이클=자동 / is_a·part_of 애매=locked 수작업 / 그 사이=LLM).
- **근거:** 모순 감지에 LLM 불필요((from,to) 비교라 코드로 충분). 운영 맵은
  precision-first — 충돌은 맵에 안 박고 대기열(`_07.conflicts.json`)에만.
- **산출물:** `{id}_07.conflicts.json` (감지된 충돌 = 대기열. ReAct 단독은 `[]`).

### 08 통합 — Integration (구 09)

- **역할:** 판정 산출물(05 노드, 06 엣지, 07 충돌)을 실제 맵에 쓴다. **유일하게 맵을
  건드리는 단계.** 05·06·07은 판정만, 08만 반영 — 판정/반영 분리가 핵심 구조.
- **노드:** merge=정의 append(text 완전일치 dedup, mastery 보존) / new=노드 추가
  (id=05 name, type=null, mastery=0, locked=false). `locked` 노드는 자동 갱신 차단.
- **SELECT (유일한 LLM 호출):** multi-def 노드의 정의 더미 → 깨끗한 단수 1개. 날짜
  가중치는 미구현("최신=정답" 아닐 수 있어 데이터 보고 결정).
- **엣지 = 누적 (덮어쓰기 버그 수정).** 이전엔 새 엣지로 맵 엣지를 통째 덮어써 두 번째
  논문이 첫 논문 엣지를 지웠다. 이제 **기존 맵 엣지 + 새 엣지** 누적 + `(from,rel,to)`
  dedup. 충돌 분리는 **보수적 = 기존 신뢰**:
  - 기존 맵 엣지 → 항상 유지 (미검증 새 도전자 때문에 검증된 걸 안 내림).
  - 충돌난 새 엣지 → 맵 제외, `_07.conflicts.json` 대기열로 보류.
  - 중복·충돌 아닌 새 엣지 → 맵에 추가.
  - (동일 논문 내 새 엣지끼리 충돌 = 우선순위 없음 → 둘 다 보류.)
- **산출 둘:** `_08.integrated.json`(08a, 정의 더미 + 충돌 스냅샷, 디버그) /
  `knowledge_map.json`(08b, 정의 단수, 운영). 충돌은 운영 맵에 안 박음(precision-first).
- **배제한 길:** ① 충돌 시 "둘 다 넣고 로그만" → 맵에 모순 남아 폐기. ② "기존도 같이
  내림" → 검증된 걸 미검증 도전자 때문에 내리는 건 공격적, 폐기. ③ conflicts를 운영
  맵에 저장 → 안 함.
- **실행:** `uv run python -m agents.agent_08_integration --paper <id>`(스텁, 파일
  안 씀·결정성만 검증) / `--run`(실제 SELECT + 저장).

## 3. 관통하는 원칙

1. **맵은 지형도, 디테일 저장소 아님** — 수치·세팅 버림. 디테일은 사용자가 나중에 파고듦.
2. **외부지식 절대 금지** — 모든 정의는 논문 근거만. LLM 사전지식 채우기 금지(할루).
3. **정의문으로 비교 (KARMA는 이름)** — AI 개념엔 도메인 임베딩 모델이 없어서. 이 선택이 우리만의 문제·차별점을 만듦.
4. **거친 한 바퀴 먼저** — 인프라 과잉투자 경계. 관련성 컷·파서 추상화·인자화는 실제 블로커 생길 때.
5. **이름값 아닌 적합성으로 도구 선택** — status(LLM 자기판정) 폐기, BKT 폐기와 같은 결.
6. **정보 안 버리고 판단 가능한 곳에서 결정** — 정의 다중 보관 → align이 best 선택.
7. **검증된 라이브러리 직접 안 짬** — pymupdf4llm 등.
8. **enrichment** — 빈약/오염된 정의는 버그가 아니라 "아직 본진 논문 안 옴". 논문 쌓이며 채워짐.
9. **착수 순서 ≠ 단계 번호.** CRA(07)는 *맵 전체와 충돌*이 일인데, 맵에 논문 2편+
   쌓여야 비로소 일거리가 생긴다(그래서 `detect`만 만들고 `resolve`는 데이터 보고
   미룸). 08 통합이 먼저 실제 블로커였다. "결정은 실제 블로커에 붙인다."

## 5. 미결 / 다음 과제

- **08 통합 = 구현 완료.** 05 노드 + 06 엣지 + 07 충돌을 `knowledge_map.json`에 반영.
  엣지 누적 버그 수정 + 07 CRA 충돌 감지 연결. ReAct 단독 회귀 = 23노드/1엣지/0충돌.
- **07 `resolve` = 다음 과제.** 충돌 *감지*는 완료, *해소*는 미구현(스텁). 실제 충돌
  사례(두 번째 논문 투입 시)가 쌓이면 방식 확정.
- **두 번째 논문(ToT 등) 투입 = 다음 검증.** 기존 엣지 생존(누적) + 새 엣지 추가 +
  충돌 시 대기열 격리를 실데이터로 확인.
- **2종(is_a/part_of) 충분성: 실측으로 닫힘** (위 06 실측). a priori 논쟁 종료.
- **HITL:** human-edited 노드 자동 덮어쓰기 금지(`locked` 플래그). 통합에 가드는
  들어갔으나 현재 전 노드 `locked=false`라 실효 검증은 수동 잠금 사례 생길 때.
- **enrichment 검증:** 두 번째 논문(CoT 논문 등)을 통합하면, ReAct만 있을 땐 안 보이던
  is_a가 공통 상위 개념을 통해 붙기 시작하는지 확인 (빚 ②와 묶임).
