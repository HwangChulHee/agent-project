# 학습 회고

각 단계를 마치면서 정리하는 학습 노트. 시간이 흐른 후 다시 봤을 때
"무엇을 배웠는지, 어디서 막혔는지, 어떤 결정이 좋았는지" 회상할 수 있도록 기록.

## 1~2단계 회고

### 가장 가치 있었던 깨달음

**1. LLM은 텍스트 생성기일 뿐**

도구 "선택"은 사실 약속된 텍스트 형식을 모델이 따라 쓰는 것 + 우리 코드가 그걸 파싱해서
진짜 함수를 실행하는 것이다. 이 본질을 1단계에서 직접 코드로 보면서 명확해졌다.

Function calling으로 갈 때도 본질은 같다 — 단지 "약속된 형식"이 자유 텍스트에서
JSON으로 바뀌었을 뿐. 새로운 마법이 아니다.

이걸 알면 GPT, Claude, Gemini, MCP, 또는 미래의 어떤 형식이든 빠르게 적응 가능하다.

**2. 메시지 누적이 "에이전트의 기억"**

LLM은 stateless다. 매 호출마다 모든 컨텍스트를 다시 보내야 한다.
`messages.append(...)`로 메시지가 누적되고, 다음 LLM 호출 시 전체가 전달된다.
이게 사이클 1에서 했던 일을 사이클 5의 LLM이 알 수 있는 유일한 메커니즘.

여기서 두 가지 미래 문제가 보였다:
- 컨텍스트 한도 (32K) 도달 → 메모리/요약 필요
- 토큰 비용 누적 → 캐싱·압축 필요

이게 3단계(메모리 + RAG)의 동기가 됐다.

**3. 도구 설계가 에이전트 동작을 좌우한다**

T4의 9 사이클 헛돌이의 진짜 원인은 모델이 아니라 wiki 도구가 5문장 요약만 줬기 때문이다.
모델은 도구의 한계 안에서 최선을 다했고, 결국 자기 지식으로 우회했다.

**LLM 자체의 능력만큼 도구의 인터페이스가 중요하다.** 좋은 도구는:
- 충분한 정보를 줌 (요약 길이, 추출 깊이)
- 명확한 description (모델이 언제 쓸지 판단 가능)
- 에러 메시지가 친절 (모델이 다음 행동 결정 가능)

**4. 인터페이스 개선 ≠ 본질 해결**

Function calling으로 가서 T4의 형식 어김 문제는 사라졌다. 사이클 수도 9→5로 줄었다.
하지만 **wiki 도구의 정보 부족은 그대로**. 인터페이스를 바꾼다고 본질 문제가 해결되진 않는다.

다음 단계들이 본질 문제를 다룬다:
- RAG: 더 풍부한 검색
- MCP: 표준화된 강력한 도구

### 좋았던 결정들

**1. 텍스트 파싱 ReAct를 먼저 구현하기**

처음부터 Function calling으로 갔다면 "왜 이게 표준이 됐는지" 몰랐을 거다.
1단계의 형식 깨짐(사이클 3의 Action 두 번 출력)을 직접 보고 나서야 2단계의 가치를 안다.

학습엔 항상 "왜?"의 동기가 필요하다. 옛 방식을 직접 만들어보는 게 그 동기를 준다.

**2. 양자화 모델로 시작**

26B AWQ 4-bit 모델로 학습하면서 양자화의 영향을 직접 관찰할 수 있었다:
- 일반 작업: 영향 거의 없음
- 시스템 프롬프트의 미세 가이드: 가끔 무시 (`"Before calling a tool, briefly explain..."`)
- Function calling 형식: 안정적

학습 중에 풀 정밀도 모델을 써야 할 이유는 거의 없다. 비용·속도 trade-off에서
양자화가 유리.

**3. wiki 도구를 재구현한 것**

원래 `wikipedia` PyPI 패키지를 쓰려 했는데 User-Agent 정책 강화로 깨졌다.
`requests`로 REST API 직접 호출로 재구현한 게 학습 가치가 컸다:
- 친절한 추상화일수록 깨졌을 때 디버깅이 어려움
- 외부 API 직접 호출 패턴은 에이전트 개발에서 매번 만남
- "공식 문서를 맹신하지 말고 항상 실제 응답 확인" 습관

### 막혔던 곳, 함정

**1. vLLM nightly + Gemma 4 + transformers 버전 충돌**

처음 설치할 때 vLLM 안정 버전이 transformers를 4.x로 핀해서 Gemma 4 인식 실패.
nightly 빌드 + cu129 인덱스 + `--index-strategy unsafe-best-match`로 해결.

**교훈**: 최신 모델은 라이브러리 버전 충돌이 거의 확실. nightly 빌드 + uv의 멀티 인덱스
처리가 도움.

**2. vLLM Gemma 4 멀티모달 초기화 함정**

Gemma 4는 멀티모달(텍스트+이미지) 모델이라 vLLM이 이미지 처리 인코더 예산을 자동 계산하는데,
그게 기본 `max-num-batched-tokens=2048`을 초과해서 실패. `--max-num-batched-tokens 8192`로 해결.

**교훈**: vLLM 에러 메시지를 잘 읽으면 해결책이 명시되어 있다. 추측 X.

**3. `wikipedia` PyPI 패키지의 User-Agent 정책 위반**

처음엔 User-Agent 차단이라고 추측했는데, REST API는 통과하고 구식 MediaWiki API만
차단되는 거였다. `wikipedia` 패키지는 후자를 써서 깨짐.

**교훈**: 추측 후 검증. 첫 진단이 부분적으로만 맞을 수 있음.

**4. `reasoning_content` vs `reasoning` 필드명**

Gemma 4 thinking 모드 활성화했더니 `reasoning_content`가 비어있어서 동작 안 한 줄 알았다.
실제로는 vLLM nightly가 OpenAI 컨벤션 따라 `reasoning` 필드명을 씀.

**교훈**: 모델·라이브러리 문서를 맹신하지 말고 응답을 직접 확인 (`message.model_dump_json()`).

**5. 가상환경 활성화 상태에서 다른 프로젝트 디렉토리로 이동**

`step2-function-calling`에서 `uv init`했는데 `step1-react`의 venv가 활성화돼있어서 시스템
Python을 가져옴. `deactivate` + 명시적 `uv sync --python <경로>`로 해결.

**교훈**: 프로젝트마다 셸 깨끗이 닫고 시작.

### Gemma 4 26B-A4B AWQ의 실력 평가

학습 시작 전 우려했던 것들이 대부분 기우였다:

| 우려 | 실제 |
|---|---|
| 양자화로 tool calling 정확도 ↓ | 정상, 한 번도 JSON 어김 없음 |
| MoE 모델이 ReAct 못 할 것 | 잘 함, thinking 모드까지 활용 |
| 한국어 답변 약할 것 | 충분히 좋음 |
| 자기 한계 인지 못 할 것 | T4 사이클 5에서 명시적으로 인지 |
| Thinking 모드가 도구 호출과 충돌 | 잘 분리됨 |

결론: 학습용엔 31B Dense 받을 필요 없음. 26B MoE 충분.

### 다음 단계 가는 길

**3단계 (메모리 + RAG)로 가는 동기**:
- T4의 6번 wiki 검색 중 3번이 결과적으로 같은 페이지 가져옴 → 캐싱 필요
- 5문장 요약은 멀티스텝 추론엔 정보 부족 → 벡터 검색으로 풍부화
- messages 누적 → 컨텍스트 한도 도달 위험 → 요약/압축 필요

**이 단계에서 답을 못 한 질문들** (3단계 이후로 미룸):
- 도구 호출이 동일한데 결과가 다른 경우 (시간 의존) 캐싱 어떻게?
- 여러 사용자가 동시에 쓰는 에이전트의 메모리는 어떻게 격리?
- 에이전트가 학습 결과를 다음 세션에 가져갈 수 있나? (지속적 메모리)

이런 질문들이 다음 단계 진입 시 동기로 작용할 것.
